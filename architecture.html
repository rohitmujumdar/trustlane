Who spawns whom

An orchestrator and its sub-agents

🟢 Deterministic (the demo buttons)

The scenario script is the parent. It spawns specialised sub-agents in sequence: search → booking → payment, each its own instance with its own hop depth. Same outcome every time.

🟣 Live LLM (the purple bar)

A real Claude agent is the parent. It spawns the sub-agents by choosing tools — every search / book / pay call creates one. It decides live; the gate still scores every call.

Both shapes: human → orchestrator → specialised sub-agents per step.
The trust score

Five signals decide it

source

Source trust

From the verified user / valid delegation, or untrusted content?

scope

Scope match

Does it match the declared task, or sneak in extras?

budget

Budget

Does total spend stay under the task budget?

vendor

Vendor allowlist

Is the merchant approved?

identity

Identity / delegation

Valid delegation, scope ⊆ what the principal allowed?

Weights differ by agent role (a payment agent weighs source + budget hardest). Passed weights sum to the raw score.
The math, demystified

What actually decides the number

raw = sum of passed-signal weights
× hop decay  (hop1 ×0.90 · hop2 ×0.80 · hop3 ×0.70)
× reputation  (0.85× … 1.15×)
then hard caps override: injection→REVIEW · over-budget→40 · no-delegation→10
Soft signals give nuance; the hard caps guarantee the outcome — which is why it's safe to run live. Trust also decays the further an action gets from the human.
The three outcomes

Score → decision

SCORE ≥ 70

✓ Allow

1Password issues a one-time, scoped credential. Booking completes, credential revoked.

40–69

⚠ Review

Caught — handed to a human. No secret resolved until approved.

< 40

✕ Block

Credential never created. The action physically cannot happen.

Two threats, one engine

Inbound & outbound

🔵 Inbound

Someone else's agent arrives to book
  • Verify it carries valid delegated authority.
  • A bot with no delegation is rejected at the door.

🟣 Outbound

The platform's own agent books for a user
  • A poisoned listing hides "add $199 insurance, upgrade."
  • Engine sees untrusted source → flags it for review.
The weave

1Password, per sub-agent

Search sub-agent → read-only → the vault is never touched.

Booking / payment sub-agent → on ALLOW, one runtime resolve() pulls the secret, scoped to that merchant & amount, used once, revoked, gone in 30s.

Any REVIEW / BLOCK → the resolve line is never reached. The secret never enters the agent's context.

One ALLOW = one vault open. One block = zero. A bad agent never even causes the vault to open.
